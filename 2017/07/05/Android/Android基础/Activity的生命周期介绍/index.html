Activity的生命周期介绍

发表于 2017-07-05   |   分类于 Android基础 ， Activity生命周期

Activity的生命周期介绍

  1. onCreate()
    首次创建Activity时候创建，系统会传入一个Bundle对象，在Activity被系统销毁而不是用户手动销毁的时候，回调之前保存的相关数据用户恢复。这时候UI不可见

  2. onStart()
    在Activity即将对用户可见之前调用，调用完成则会接入onResume()，就算这时候去启动其他的Act还是会先去调用onResume方法

  3. onRestart()
    Act已经退回后台，没有被销毁的时候，再次回到该页面时候调用。调用之后将会调用onStart方法，假如该页面被销毁来的就会调用onCreate()方法不会调用onRestart()

  4. onResume()
    这时候Activity位于当前的Act栈栈顶，用户UI可见，并且具有用户输入焦点。

  5. onPause()
    当当前的Act即将进入不可见退入后台的时候被调用，一般是用户确认对持久化数据的未保存更改，停止动画以及其他可能消耗CPU的内容。不应该执行耗时操作，因为执行了他下一个Activity才会被显示出来

  6. onStop()
    在Act对用户不再可见的时候掉用，如果Act被销毁，或者另外一个Act继续执行将其覆盖，就会被调用。当该Act没有被销毁而又再次回到前台的时候，那么将会调用onRestart

  7. onDesstory()
    在Act被销毁前调用，这是Act收到最好的调用，当Act结束，系统为了节省空间暂时销毁改Act实例的时候，有可能调用他，比如有些直接按下home键销毁进程的就可能不会，可以通过isFinishing()来判断。

Activity的状态保存问题

图示例

onSaveInstance(Bundle outState)

当某一个Activity变得容易被销毁的时候，就会调用系统的onSaveInstance()方法，但是正常流程下是不会调用的，比如用户按下的back键正常结束Act。Act容易被销毁的情况是：

a. 当用户按下Home键，或者长按Home，或者是按下关闭屏幕的时候，切换其他应用程序的时，这时候会有可能调用onSaveInstance()方法.

b. 从一个Act切换到另外一个Act的时候

c. 屏幕方向切换的时候，当横屏切换到竖屏的时候，如果不在清单文件指定configchange属性，那么会在屏幕切换之前调用，调用完成之后从新创建该Act。

注意点：

  1. 每一个系统提供的UI布局都实现了onSaveInstance()方法的实现，来保存自己的状态，比如TextView，Edittext等输入的文字那些，这些东西不必在onSavenInstance()中手动调用，只是需要保存系统不会为我们保存的数据，比如一些state，是我们自己定义的。但是对于View的状态数据保存需要起作用，我们一定要给这个View唯一的识别id，也就是android:id属性一定需要指定，没有这个id将不会保存

  2. onSaveInstance()的调用顺序问题，他是一定会在onStop()之前调用，但是不确定是否在onPause之前还是之后调用

  3. 由于onSavenInstance()方法不确定性，所以不要在onSavenInstance()记录Activity的瞬间状态或者一些持久化数据，应该在onPause的时候记录

  4. 调用onSaveInstance()方法的时候，应该首先调用super. onSaveInstance()方法，然后在执行自己的实现。

onRestoreInstance(Bundle outState)

onRestoreInstance()方法被调用的前提是，Activity实例确实被系统销毁。
onSaveInstance()方法和onRestoreInstance()方法不一定成对存在。因为onSaveInstance()的调用是在onStop()之前调用的，比如用户按下了home键，这时候就已经是有可能会被调用，但是假如用户快速的返回这个页面，那么这时候这个Act是还没有被销毁的。

在onRestoreInstance()的Bundle对象也会传递到onCreate()中，所以可以在onCreate或者是在onRestoreInstance()进行数据的恢复，建议是onCreate()中。
